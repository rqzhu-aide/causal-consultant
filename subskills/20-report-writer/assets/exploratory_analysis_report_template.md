# Gate-Not-Ready Exploratory / Progress Analysis Report Template

Use this flexible template when analyzable data exist and the user wants a report-like artifact while the foundation or production gate is not ready for final handoff. This report is for learning, debugging, model exploration, production progress, diagnostics, and hypothesis generation. It is not a final causal report.

The default deliverable is a reproducible source report plus rendered HTML:

```text
artifacts/
  [descriptive-name]-exploratory.ipynb   # or .qmd / .Rmd
  [descriptive-name]-exploratory.html
  figures/
  tables/
```

The source file is the reproducible artifact. The `.html` is the reading and sharing artifact. If render is infeasible, label the source report as a draft or analysis source rather than a completed shareable report.

## Required Claim Boundary

The report must clearly state the gatekeeper state near the beginning:

```yaml
foundation_gate.status: ready | exploratory | blocked
foundation_gate.can_support_causal_commitment: true | false
production_gate.status: not ready | blocked
production_gate.can_handoff_to_report_writer: false
analysis.route_commitment_status: user-directed | ready | committed | exploratory | blocked
analysis.claim_strength: exploratory | associational | descriptive | cautious causal
```

Do not present any estimate as a final validated causal effect. If foundation is not ready, use language such as "exploratory estimate", "model-based first pass", "association under this specification", or "not validated as causal." If foundation is ready but production is not ready, use language such as "first-pass result", "diagnostic result", or "not yet report-ready." If the user needs a final causal report, say what must change before the gate-ready report template can be used.

## Template Instructions

Write the source report as a short technical report, not a checklist. Use numbered sections, but adapt section titles, order, and length to the project. Keep the same general spine as the gate-ready report so users can compare versions easily. Explain what was tried, what the data/model appear to show, why the artifact is not yet gate-ready, and what next check would most improve the claim.

Use bullets or tables when they improve clarity. Keep the report lean by default. The exploratory lane should help the user learn quickly, not bury them in final-report polish.

## Flexible Report Spine

Use this spine as a starting point. Merge, split, rename, or reorder sections when the project needs it.

```markdown
# [Project-Specific Exploratory / Progress Report Title]

## 1. Summary And Claim Boundary

[Summarize the user's request, the data inspected, the model attempted, the main pattern or estimate if available, and the reason this artifact is not a gate-ready causal report.]

[Include a short claim-status paragraph. If foundation is not ready, say this is exploratory/user-directed analysis and estimates are model-based first-pass outputs, not validated causal effects. If foundation is ready but production is not ready, say this is a progress or diagnostic report and results are not final report conclusions.]

## 2. Question, Data, And Design

[Describe the user request in practical terms. Name treatment/exposure, outcome, comparator, population, timing, target estimand, and candidate design if known. State which of these are uncertain, incomplete, provisional, or already foundation-ready.]

[List the data files or sources actually inspected. Note missing files, inaccessible fields, empty data, nonconstructible variables, or unresolved measurement/timing problems.]

## 3. Data Readiness And Analysis Specification

[Describe row unit, missingness, timing/leakage risks, support or overlap, treatment/outcome availability, and any minimal preprocessing.]

[Describe the attempted model, design, method owner, packages, estimand, adjustment or matching strategy if any, and planned or missing diagnostics. If useful, include a compact equation in the appendix rather than the main text.]

[If a method/job subskill or production-loop ledger exists, name it briefly as provenance for the attempted analysis. Do not describe it as production readiness unless the gatekeeper fields support that.]

## 4. Results And Diagnostics

[Present descriptive summaries, exploratory or first-pass estimates, intervals, figures, model outputs, and diagnostics generated by executed code or verified artifacts.]

[Use cautious interpretation. For pre-foundation work, prefer "under this model..." or "in this first pass..." rather than causal wording. For post-foundation/pre-production-gate work, say which diagnostics or sensitivity checks still limit report readiness.]

## 5. Interpretation And Next Step

[State what can be learned, what cannot be claimed, and how strong the evidence is. Separate effect-estimation-style output from action recommendations.]

[Name the blockers or missing pieces: missing design information, constructability issues, timing ambiguity, unverified assumptions, failed diagnostics, weak support/overlap, missing sensitivity checks, unresolved confounding, or lack of production review. End with the smallest next check or user decision that could move the artifact toward gate-ready reporting.]

## 6. Reproducibility Appendix

[Record source report path, rendered HTML path, package versions, seeds, code paths, saved figures and tables, and any environment or rerun notes needed for audit.]

### Optional Analytic Specification

[When useful, include a lean mathematical specification here. Define outcome, treatment/exposure, covariates, unit/time indices, estimand, and model equation. Keep it compact and label it as the exploratory specification.]
```

## Required Coverage

Every gate-not-ready exploratory/progress analysis report must cover these elements somewhere in the report:

| Required element | What must be covered |
|---|---|
| Report scope | What the user asked to explore or review, what limits were acknowledged or recorded, and why the work is not a gate-ready causal report. |
| Gatekeeper state | Route commitment status, claim strength, foundation gate status, causal-commitment flag, production gate status, and handoff flag. |
| Data provenance | Data files or sources inspected, unavailable files, access limits, exclusions, and whether data were user-provided, workspace-inspected, or computed from authorized sources. |
| Question and design | Treatment or exposure, outcome, comparator, population, timing, target estimand, candidate design, and which parts are uncertain, provisional, or already foundation-ready. |
| Data checks | Row unit, treatment and outcome availability, missingness, timing/leakage risks, support or overlap, measurement issues, and empty or nonconstructible inputs. |
| Analysis specification | Method attempted, packages used, adjustment or matching strategy if any, model specification, diagnostics planned or missing, and why this was a reasonable report-not-ready analysis. |
| Results provenance | Tables, figures, model outputs, estimates, intervals, and diagnostics generated by executed code or copied from verified artifacts. |
| Claim boundary | What can be learned, what cannot be claimed, why the artifact is not gate-ready, and which blockers matter most. |
| Upgrade path | The smallest next checks or user decisions that could move the project toward gate-ready analysis. |
| Reproducibility | Source report path, rendered HTML path, package versions, seeds, code paths, saved figures/tables, and environment or rerun notes. |

## Code And Output Rules

- Results, diagnostics, plots, and tables must come from executed code, verified artifacts, or explicitly labeled user-provided outputs.
- Do not invent sample sizes, estimates, uncertainty intervals, p-values, diagnostics, robustness checks, or table values.
- If a code block uses illustrative or simulated data, label it clearly and do not present it as project evidence.
- If data are unavailable, empty, inaccessible, or not constructible, do not write an exploratory results section with fake numbers; switch to a planning/communication memo or executable code scaffold.
- Keep sensitive raw rows, direct identifiers, secrets, and small-cell details out of the rendered report unless explicitly approved and necessary.

## Claim Language Examples

Prefer:

- "This exploratory model estimates..."
- "Under this first-pass specification..."
- "The fitted model shows an association..."
- "This result is useful for prioritizing diagnostics, not for a causal conclusion."
- "The current data do not yet support a validated causal claim."

Avoid:

- "The causal effect is..."
- "The intervention caused..."
- "The analysis proves..."
- "The result is policy-ready..."
- "The report is final..."
