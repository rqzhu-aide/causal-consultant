# Gate-Ready Reproducible Analysis Report Template

Use this flexible template when analyzable data exist, the analysis is run in code, and the workflow has enough gate support for an effect-estimation report. This is the source-plus-HTML version of a gate-ready causal report. The default deliverable is a reproducible source report plus rendered HTML:

```text
artifacts/
  [descriptive-name].ipynb   # or .qmd / .Rmd, depending on the analysis language
  [descriptive-name].html
  figures/
  tables/
```

The source file (`.ipynb`, `.qmd`, `.Rmd`, or equivalent) is the reproducible analysis artifact. The `.html` is the reading and sharing artifact. If figures, tables, model objects, or diagnostic outputs are saved separately, link them from the report and index them in the project artifacts.

Use `exploratory_analysis_report_template.md` instead when any gatekeeper field does not support final handoff and the user still wants a first-pass, diagnostic, progress, or exploratory report.

## Template Instructions

Write the source report as a technical report, not as a checklist. Use numbered sections, but adapt section titles, order, and length to the project. Use coherent paragraphs to explain the causal question, data, design, results, diagnostics, and interpretation. Use bullets or tables only when they improve clarity.

Before writing, confirm the gatekeeper fields support this lane:

```yaml
foundation_gate.status: ready
foundation_gate.can_support_causal_commitment: true
production_gate.status: ready
production_gate.can_handoff_to_report_writer: true
analysis.route_commitment_status: ready | committed
analysis.claim_strength: causal | cautious causal | associational | descriptive | exploratory
```

The claim strength can still be cautious, associational, descriptive, or exploratory, but the route and production materials must be ready enough for Report Writer handoff. If any gatekeeper field is not compatible with handoff, switch to the exploratory template or a planning/communication memo.

The report may be short for a simple randomized experiment and much longer for an observational, longitudinal, discovery, survival, IV, or synthetic-control project. The section spine is flexible, but the report must cover the required elements listed below somewhere in the source report and rendered HTML.

## Flexible Report Spine

Use this spine as a starting point. Merge, split, rename, or reorder sections when the project needs it.

```markdown
# [Project-Specific Title]

## 1. Summary And Claim Status

[Write a brief technical summary of the user question, the data used, the design or method, the main result if available, the current claim strength, and the most important limitation. State that this is the gate-ready report lane and name any cautious or non-causal claim-strength limits.]

## 2. Question, Data, And Design

[Explain the causal question in prose. Name the treatment or exposure, outcome, comparator, population, timing, and target estimand. Describe the data sources that were actually inspected and the design logic that makes the analysis plausible or fragile.]

[Use a compact table only if it improves clarity, for example to summarize treatment, outcome, comparator, timing, and data source.]

## 3. Data Readiness And Analysis Specification

[Describe row unit, missingness, treatment and outcome availability, timing or leakage risks, support or overlap, measurement issues, and any exclusions. Then describe the selected route or method owner, packages used, estimand, adjustment or matching strategy when relevant, and planned diagnostics.]

[Describe the analytic specification in plain language. Include a compact equation in the main text only when it helps the intended audience understand the design or result. Otherwise, place the lean mathematical specification in the Reproducibility Appendix under "Analytic Specification".]

[Place data-loading, cleaning, and setup code near this section. Outputs should support the prose.]

## 4. Results And Diagnostics

[Present descriptive summaries, estimates, intervals, figures, model outputs, and route-specific diagnostics generated by executed code or verified artifacts. Explain how each important diagnostic changes interpretation.]

[If diagnostics are not run, fail, or are deferred, state that directly and explain the consequence for claim strength.]

## 5. Interpretation And Next Step

[State what can be claimed, what cannot be claimed, how strong the evidence is, and what action recommendation limits remain. End with the most useful next analysis, decision, or user-facing follow-up.]

## 6. Reproducibility Appendix

[Record source report path, rendered HTML path, package versions, seeds, code paths, saved figures and tables, and any environment or rerun notes needed for audit.]

### Analytic Specification

[When useful, include a lean mathematical specification here. Define the outcome, treatment or exposure, key covariates, unit/time indices, estimand, and main model or design equation. Keep it compact; avoid derivations unless the user asked for a technical or academic report.]
```

For complex projects, expand the middle sections. For short projects, combine related material while preserving the required coverage.

## Required Coverage

Every gate-ready data-backed reproducible report must cover these elements somewhere in the report:

| Required element | What must be covered |
|---|---|
| Data provenance | Data files or sources inspected, access date when useful, exclusions, unavailable files, and whether data were user-provided, workspace-inspected, or computed from authorized sources. |
| Causal question and design | Treatment or exposure, outcome, comparator, population, timing, target estimand, candidate design, and why the design matches or fails the user's decision question. |
| Data readiness | Row unit, treatment and outcome availability, missingness, timing and leakage risks, support or overlap, measurement issues, and whether the data can support the intended claim. |
| Analysis specification | Selected route or method owner, packages used, estimand, prose model/design description, optional compact equation, adjustment or matching strategy when relevant, and diagnostics planned. |
| Results | Tables, figures, model outputs, estimates, intervals, and descriptive summaries generated by executed code or copied from verified artifacts. |
| Diagnostics and sensitivity | Route-specific checks, failed checks, deferred checks, robustness or sensitivity results, and how each affects claim strength. |
| Interpretation and limits | What can be claimed, what cannot be claimed, action recommendation limits, external validity cautions, and the most useful next step. |
| Reproducibility | Source report path, rendered HTML path, package versions, seeds, code paths, saved figures/tables, and any environment or rerun notes needed for audit. |
| Gatekeeper state | Foundation and production gate status, route commitment status, claim strength, and any deferred diagnostics or assumptions that limit the report. |

## Code And Output Rules

- Results, diagnostics, plots, and tables must come from executed code, verified artifacts, or explicitly labeled user-provided outputs.
- Do not hand-write numeric results into prose unless their source is visible in the source report or linked artifacts.
- Keep data-loading, cleaning, estimation, diagnostics, and rendering steps auditable.
- If a result is missing, unavailable, or not run, say so and explain how that limits the claim.
- If a code block uses illustrative or simulated data, label it clearly and do not present it as project evidence.
- Keep sensitive raw rows, direct identifiers, secrets, and small-cell details out of the rendered report unless explicitly approved and necessary.

## Route-Aware Package Guidance

Use packages that match the route or method subskill actually being used. Do not load broad causal libraries just to make the report look causal. Mention only the language stack that is actually used unless the report is comparing implementation options.

| Candidate route | R package examples | Python package examples | Typical use |
|---|---|---|---|
| Randomized experiment or A/B test | `estimatr`, `broom`, `DeclareDesign` | `pandas`, `numpy`, `scipy`, `statsmodels` | Balance, difference in means/proportions, confidence intervals, regression adjustment. |
| Regression / adjustment / weighting / doubly robust | `MatchIt`, `WeightIt`, `cobalt`, `grf`, `DoubleML` | `statsmodels`, `sklearn`, `DoubleML`, `econml`, `dowhy` | First-pass adjustment, balance/overlap, nuisance models, DR estimation, sensitivity framing. |
| DiD / event study | `fixest`, `did`, `broom` | `statsmodels`, `linearmodels`, `plotnine` or `matplotlib` | Panel setup, fixed effects, event-study plots, pre-trend checks. |
| Instrumental variables | `AER`, `ivreg`, `fixest` | `linearmodels`, `statsmodels` | First stage, reduced form, IV estimate, weak-instrument diagnostics where available. |
| Regression discontinuity | `rdrobust`, `rddensity` | `statsmodels`, route-specific RD utilities if available | Running-variable checks, bandwidth analysis, local models, density/balance checks. |
| Synthetic control / time series | `Synth`, `tidysynth`, `gsynth` | `statsmodels`, suitable synthetic-control packages if available | Pre-period fit, placebo checks, donor weights, time-series diagnostics. |
| Survival / competing risks | `survival`, `cmprsk`, `riskRegression` | `lifelines`, `scikit-survival` when available | Time-to-event setup, censoring summaries, survival or competing-risk analysis. |
| Causal discovery | `bnlearn`, `pcalg` | `networkx`, `pgmpy`, `causal-learn`, `dowhy.gcm` when appropriate and available | Graph learning, graph plotting, stability summaries, exploratory graph diagnostics. |

If the route is unclear, include a short provisional package note instead of a package list:

> Package choices are provisional because the design route is not selected yet. The next step is to clarify assignment, timing, comparator, outcome, and whether the data support an effect-estimation route or only descriptive/exploratory analysis.
